ShardKV Review

Created2025-03-30|Updated2025-03-30|Distributed System
|Post View:

代码执行流程

首先由config创建多个配置组，每个组选出一个Leader。

ShardCtrler

ShardCtrler中定义了分片的个数（10），在加入或减少组时，能自动进行shard的分配平衡。分片平衡上我使用了贪心策略。每次都将拥有Shard最多的组的shard分配给最少的组，直到所有group的shard数目平均或者相差1。这个策略能够保证负载均衡的同时，尽量让分片的移动数量变少。类似的方法还有一致性哈希，但是可能会有分片数目不均的可能，不满足实验要求。

实现的时候用了好几个循环还有双层循环，但是分片一共就10个，所以先不用考虑算法效率的问题了。

Operation

Shardctrler支持Query、Join、Leave、Move四种操作，其中Move为测试所用，将Shard从一个组移动到另一个组。

Query主要用于raft组查询最新的配置，也能够查询先前的配置。如果查询参数为-1或者大于了已知最大的配置号，那么就返回最新的配置。Join和Leave用来添加组或者移除组。

Client/Server

Client端的实现都很类似，遍历所有的服务器找到是Leader的那个，然后暂存leader编号。在发送请求时要带上自己的ClientId和seq操作序列号。ShardKV的客户端还要先根据操作key映射到指定的gid上，然后向组中的Leader发送请求，如果失败就尝试通过Query更新配置，然后重试。Client端会一直重试，直到操作成功。

Server端的各种请求响应函数其实都可以包装成一个函数，它们的作用都是将客户端发送来的请求包装为Command，Command通过Op种类进行区分，然后将这个Command通过Start函数提交给Raft进行复制。最后通过通知通道等待applier发送指令的执行情况，将信息返回给客户端。响应函数会为每个指令创建一个Notify通道，当applier收到raft提交的指令后，会获取相应的通道，在执行完指令后通过该通道通知响应函数。下图展示了Server内部指令相关的运行流程，总体上呈现一个逆时针运行顺序。

Shard转移

Shard转移是Lab5的一个难点，我认为这部分的核心在于对Shard的状态的处理上：

只要本组当前的配置中这个shard属于自己，那么就将其设置为Serving；如果属于但是数据还在别的组手里，就将其设置为Pulling；同样，不属于自己的Shard，也应该准备将其发送出去，设置为Sending；至于GCing状态，也是最难以理解的一个状态，它的意思并不是这个分片需要被删除，而是需要向上一任拥有这个分片的组发送删除信号，让该组删除这个分片。GCing状态下的分片是能够正常工作的，删除请求收到成功的反馈后就可以将其设置为Serving。

通过这几种状态设定，可以实现未受配置更新影响的分片在配置更新期间继续工作，不用封锁所有的分片实现更新

具体的配置更新和分片转移由几个协程定期执行。

listenConfig

listenConfig并不是单纯的获取最新配置然后应用，而是需要先检查一遍本组所有shard的状态，如果有不是Serving状态的，说明还有Shard的状态转移需要做，不能进行配置更新。此外，在使用Query查询配置时，也不能直接使用Query(-1)获取最新配置，因为在最新配置和当前配置之间可能有多个配置还未被使用到，直接获取最新配置可能会与其他组发生冲突。总结一下，为了让所有的配置都能够同步有两个基本保证：

  • 当前Shard中有不是Serving状态的不能更新配置
  • 每次查询时使用Query(currentConfigNum + 1)，而不是Query(-1)

当然只有这两个保证是完全不够的，比如将GCing状态转化为Serving时，可能出现目标组成功接收信息但是返回成功信号的过程中出现了网络故障，目标组更新了配置而未收到成功信号的组还在使用上一个配置。解决这类问题在后文提到。

listenPullingShard

使用lishtenPullingShard函数从其他组获取Shard时，需要从lastConfig中获取gid与shard对应的信息，所以需要在Server的元数据中需要存储lastConfig。然后，由于我们并不知道目标组的Leader是谁，所以采用轮询的方式，实现上采用了WaitGroup组等待轮询结束

那么顺便看一下GetShard的实现，要注意深拷贝和ConfigNum上的处理。深拷贝在后文中提到，这里主要讨论ConfigNum。如果args的ConfigNum大于currentConfigNum，说明是一个配置更新的组发来的请求，直接拒绝。如果args的configNum小于等于currentConfigNum，等于时不必多说，小于时则需要考虑一下，因为本组的对应的Shard可能并不处于Sending状态，但是为了能够让旧的配置组能够尽早更新，还是可以直接将对应Shard发送出去的。

但是我认为上述情况应该不会发生。如果有一个旧配置需要某一个Shard的话，那么在发送端处于这个旧配置的状态时该Shard肯定处于Sending状态，从Sending状态转变为Serving状态需要索取端发送Delete信号，而如果索取端索取失败了，是不会发送Delete信号的，发送端因此也不会进行配置的更新。

还要注意的一点是发送Shard时还应该将对应分片的LastOperation发送过去，防止边界条件下客户端的重复操作。

listenDeleteShard

这里跟PullingShard类似，找到所有GCing状态的shard以及它们在lastConfig中对应的组，发送Delete请求，在DeleteShard函数中，如果args的ConfigNum小于currentConfigNum，可以直接返回OK信号，让旧配置更新。如果发送请求端接收到删除成功的信号，也会向applier指令，将GCing状态的shard变更为Serving。

‍

Applier

applier的代码非常公式化，一部分处理指令，一部分处理快照。所有通过Raft进行复制的命令最终都要通过Applier应用到各个Server的状态机上。

ShardKV中的指令类型主要分为以上几种，其中EmptyLog是用来解除Leader变更时产生的活锁问题用的（空日志检测）。此外之前还用了一个UpdateShard用来更新Shard状态，不过后来发现状态更新的操作在其他三种关于配置的操作过程中都能顺便更新，也就不用专门进行状态更新了。

关于普通的Operation操作已经不想多说，做完整个项目相当于写了三遍这个部分，只需要做一些针对性的改动就行。在实现ShardKV的状态机时，我也写了一个apply函数用来处理get，put，append操作，所以applier只需要将任务下放给状态机就行，applier只需要处理状态机的返回值生成对应的reply。

深拷贝

这个Lab中涉及到map的操作都应进行深拷贝（即创建一个map将原先的值都拷贝到这个map中），map（如 Go 语言中的 map）通常是通过引用传递的。这意味着，当你将一个 map 传递给函数或其他组件时，实际上传递的是指向同一底层数据结构的指针，而不是数据的副本。

线性一致性

在 6.5840 的 KV 实验中，为了简单起见，所有操作（包括查询）都走 Raft 以保证线性一致性，但在实际工业场景中，对于只读操作不一定需要像修改操作那样通过整个 Raft 日志复制过程来实现线性一致性。实际上，工业系统中常用以下几种方法来优化只读操作，同时保证线性一致性：

  1. 基于租约的只读操作（Leader Lease）

    • 原理：如果 leader 持有一个租约（lease），保证在租约有效期间没有其他节点可以成为新的 leader，那么 leader 就可以直接从本地状态机读取数据，而不需要通过 Raft 日志提交。
    • 实现：这种方法依赖于周期性的心跳消息，leader 在收到多数节点确认后可以认为自己依然有效，然后安全地服务只读请求。这种方式可以显著降低只读请求的延迟。
    • 例子：etcd 就提供了通过租约实现的线性化只读查询，叫做“read index”或“线性化读取”。

  2. Read Index 协议

    • 原理：通过 Raft 内部的 Read Index 协议，leader 可以向集群查询当前提交的日志索引，而不必把只读操作也写入日志。
    • 实现：读操作通过获得一个“读索引”，然后从状态机读取数据。这样既能保证线性一致性，又避免了写入日志的开销。

  3. 利用全局时间戳或 TrueTime（如 Google Spanner）

    • 原理：使用外部同步的全局时间戳（如 Spanner 的 TrueTime），可以在不经过共识的情况下实现强一致的只读操作。
    • 实现：这种方法依赖于精确的时间同步，但可以提供低延迟的线性化读。

为什么只读操作不一定需要走 Raft 日志

  • 降低延迟：写操作需要经过日志复制、确认和应用，而只读操作如果能在保证线性一致性的前提下直接读取状态机，就可以大大降低延迟。
  • 减少资源消耗：只读操作如果不写入日志，就能减少 Raft 日志的增长，也避免了额外的磁盘 I/O 和网络传输开销。

总结

在实际工业场景中，通过租约或 Read Index 协议等机制，可以实现只读操作的线性一致性，而无需像写操作那样通过 Raft 日志复制。这两种方法能够保证只读请求返回最新数据，同时减少延迟和系统负载，从而提高系统的性能和可扩展性。
