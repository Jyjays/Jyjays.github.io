Task #1 TimeStamps

实现事务管理器以正确给事务分配时间戳。

当事务开始时，它将分配一个read timestamp，来自最新提交的事务的commit timestamp。在事务提交时，将分配单调递增的commit timestamp。read timestamp 确定可以通过事务读取哪些数据，commit timestamp确定事务的序列化顺序。

read timestamp 确定当前事务可以看到的版本。例如，如果我们进行read timestamp = 3的事务，则在此示例中将看到A3，B3，C2和D3。如果read timestamp = 2，它将看到A2，B1，C2。当事务开始时，read timestamp 将是最新提交的commit timestamp，以便事务能够在事务开始之前看到所做的一切。

commit timestamp 是事务提交的时间。这是一个逻辑计数器，每次提交事务都会增加一次。当DBMS修改数据库中的单元时，DBMS将使用事务的commit timestamp。

WaterMark

实现watermark机制，要求更新watermark的时间复杂度不能超过O(log n)

watermark是正在执行的transactions中的最小的read-timestamp，这一机制将会在后面的Stop-the-World Garbage Collection部分中使用到

Task #2 Storage Format and Sequential Scan

Bustub将事务数据存储在三个地方：table heap，transaction manager以及每个事务内部。table heap总是包含最新数据，transaction manager的“page version info”存储了指向下一个修改的指针，在每个事务内部，我们以称为undo log的格式存储了事务修改的元组。

  • 版本链：

    • 每个元组（Tuple）的头部包含指向旧版本的指针，形成链表结构。
    • 例如，在 Bustub 中，每个 Tuple​ 可能包含 txn_id​（创建该版本的事务 ID）和 begin_ts​/end_ts​（版本的有效时间范围）。

Tuple Reconstruction

在实现 Bustub 的 MVCC 时，Tuple Reconstruction（元组重建） 是一个关键步骤，用于根据事务的隔离级别和可见性规则，从版本链中选择正确的数据版本，并构造出事务可见的元组。

为什么需要 Tuple Reconstruction？

  • 每个事务需要看到符合其隔离级别的数据版本。
  • 由于同一数据可能有多个版本（通过版本链链接），查询时需要遍历版本链，找到对当前事务可见的版本。

函数参数包括一个undo_log vector，ReconstructTuple过程就是遍历这些undo_logs，将base_tuple转化为对应的tuple，同时需要注意处理delete操作。

Sequential Scan / Tuple Retrieval

修改Project中实现的 Sequential Scan executor，遍历table heap，检索read timestamp并通过ReconstructTuple得到需要的Tuple。

lab提示需要注意三种情况：

  1. table heap中的元组是最新数据。可以通过比较table heap tuple的 meta data 和transaction读取时间戳来知道这一点。在这种情况下，可以直接返回元组，或者直接跳过已删除元组。
  2. table heap中的元组包含当前transaction的修改。在Bustub中，有效的 commit timestamp 范围从0到TXN_START_ID -1。如果时间戳的最高数值位在表堆中设置为1，则意味着元组通过transaction修改，transaction尚未commit。我们称此时间戳为“临时时间戳”，由TXN_START_ID + TXN_HUMAN_REDABLE_ID = TXN_ID计算。 BUSTUB中的第一个事务ID是TXN_START_ID。这是一个很大很难解释的数字，因此我们可以通过取它的最高位来生成TXN_HUMAN_REDABLE_ID。在以下示例中，我们将使用txnX，其中X是TXN_HUMAN_REDABLE_ID。undo log 将永远不会包含事务临时时间戳。例如，如果当前的transaction的可读ID为3，并且它用时间戳TXN_START_ID + 3扫描基本元组，则此元组通过此事务修改，并且应直接返回给用户。否则，transaction应恢复该元组的过去版本并将其退还给用户。
  3. table heap中的元组是（1）通过另一个未承诺的事务修改，或（2）比事务读取时间戳更新。在这种情况下，您需要迭代版本链以在读取时间戳后收集所有撤消日志，并恢复元组的过去版本。

‍

  1. 表堆中的元组是最新的数据

  • 判断条件：
    通过比较表堆元组元数据中的时间戳（timestamp）与事务的读取时间戳（read timestamp）。

    • 如果元组的时间戳早于或等于事务的读取时间戳，并且元组未被删除，则该元组是最新的数据。
    • 如果元组已被删除（例如，删除标记被设置），则该元组不可用。

  • 处理方式：

    • 如果元组是最新的且未被删除，Sequential Executor 直接返回该元组。
    • 如果元组已被删除，则跳过该元组。
  1. 表堆中的元组被当前事务修改

  • 判断条件：
    在 BusTub 中，如果元组的时间戳是一个“临时时间戳”（即时间戳的最高位被设置为 1），表示该元组被某个事务修改，且该事务尚未提交。

    • 临时时间戳的计算方式为：TXN_START_ID + txn_human_readable_id = txn_id。
    • 如果该临时时间戳等于当前事务的 txn_id，说明该元组是被当前事务修改的。
    • 如果临时时间戳不等于当前事务的 txn_id，说明该元组被其他未提交的事务修改。

  • 处理方式：

    • 如果元组被当前事务修改（即临时时间戳等于当前事务的 txn_id），Sequential Executor 直接返回该元组，因为当前事务可以看到自己的修改。
    • 如果元组被其他未提交事务修改，当前事务不应该看到这个修改，需要恢复到之前的版本（具体处理方式见第 3 种情况）。
  1. 表堆中的元组被其他未提交事务修改或比读取时间戳更新

  • 判断条件：

    • 元组的时间戳是一个临时时间戳，且不等于当前事务的 txn_id，说明元组被其他未提交事务修改。
    • 元组的时间戳晚于事务的读取时间戳，说明元组是在读取时间戳之后被修改的。

  • 处理方式：
    在这两种情况下，当前事务不应该看到该版本的元组。需要执行以下步骤：

    • 遍历版本链（version chain），收集读取时间戳之后的所有 undo logs。
    • 根据 undo logs 恢复到读取时间戳时的元组版本，并返回该版本。

实际上还有一种特殊情况，那就是undo_log中最早的版本也没法匹配read timestamp，这时候只能跳过。 （把上面的条件都实现了一遍发现也过不了测试，最后发现是这种情况，不能把undo_log全找出来就进行reconstruct，还要检查一次是否达到了对应的版本）

Task #3 MVCC Executors

Insert Executor

Insert Executor 实现与Project＃3中的实现几乎相同。只需在 table heap 中创建一个新的元组即可。需要正确设置元组元数据。table heap 中的时间戳应将其设置为执行事务的临时时间戳，如Task 2.2中所述。无需修改版本链接，该元组的next version link 应为nullopt，这表明该元组没有以前的版本。您还应该将RID添加到write set中。下图是将txn9插入（d，3）的插图。

实现单线程版本的并不麻烦，lab提供了UpdateUndolink函数，将prev_version_参数置为std::nullopt即可，单线程版本暂时不需要check函数，然后在插入Tuple时将Meta数据的ts设置为事务的临时时间戳就行了。

Commit

一次只允许一次事务输入提交功能，这个部分要用到事务管理器中的commit_mutex_。Lab说明给出了如下实现步骤：

  • 获取提交锁（commit mutex）。
  • 获取 commit_ts，但在提交完成前不增加 last_committed_ts（因为提交时间戳的内容在提交结束前尚未稳定）。
  • 遍历事务修改的所有元组（通过 write set），将元组的时间戳设置为 commit_ts。
  • 将事务状态设置为 COMMITTED，并更新事务的 commit_ts。
  • 更新 last_committed_ts。

Update and Delete Executor

在此任务中，需要实现逻辑以生成撤消日志，并更新table heap元组。update 和delete executor基本上是相同的（您可以在 execution_common.cpp 中实现共享功能），update executor 将新版本的tuple放到table heap 中，而delete将is_delete标志设置到tuple。

在更新和删除之前，需要检查写-写冲突，如果一个tuple正在被一个未提交的事务修改，那么其他事务都不能修改它。另一种冲突情况是，一个事务A删除了元组并提交，另一个事务B发生在A之前且删除了同样的元组。当检测到冲突时需要将事务的状态应该被设置为TAINTED，并抛出executionException，标记SQL语句无法执行。

更新执行程序应作为Pipeline breaker实现：在编写任何更新之前，应首先将所有元组从子执行程序扫描到本地缓冲区。之后，它应该从本地缓冲区中拉出元组，计算更新的元组，然后在表堆上执行更新。

在这部分所有测试用例都是单线程，因此无需考虑更新 /删除过程中可能发生的竞争条件。检测写入冲突的唯一条件是检查基本元组元的时间戳。

检查写入冲突后，为修改创建撤消日志。对于删除，生成一个包含原始元组的完整数据的撤消日志。对于更新，生成一个仅包含修改列的撤消日志。将撤消日志存储在当前事务中。如果当前的事务已经修改了元组并为元组设有撤消日志，则应更新现有的撤消日志，而不是创建新的日志。

下面两张图片展示了Delete操作和Update操作是如何插入Undolog的。

此外在检查写-写冲突之前还需要检查是否有自我修改的情况。如果某个元组已经被当前事务修改过，那么不应该将它视为冲突。一个事务中最多为每个RID创建一个undo log，因此，如果事务两次更新元组，则需要更新已经创建好的的undo log，或者仅更新table heap。

在上图的示例中，尽管元组被多次修改，但仍保持一个undo log。还需要注意一点，从(B,5)修改为(A,5)时，尽管undo log可以变为（_,3)，但并没有这样做，undo log规定为只能添加值而不能删除值，这样更容易处理并发问题。如果该tuple是新插入的，那么在同一个事务中无论怎么再修改也不用添加undo log，如下图所示：

由于此时我们没有索引，并且仅使用RID识别所有元素，因此版本链的删除标记只可能存在在头部（在table heap中）。undo log中不会有删除标记，因为删除后不会在同一位置重新创建元组。因此，在生成撤消日志时，可以跳过很多情况。

最后总结一下，要实现这部分功能，应该：

  1. 从child exectuor获取RID
  2. 生成要更新的元组
  3. 在有自我修改的情况下，只需要更新table heap并有选择地更新undo log
  4. 否则，生成undo log，链接到链表中

在实现的过程中处理写-写冲突时，如果tuple的timestamp不是临时时间戳，那么还要将它与txn的read ts比较，如果tuple的比txn的大，那么这也是一种写-写冲突

Stop-the-World Garbage Collection

当调用GarbageCollection时，手动触发垃圾收集。仅当所有事务暂停时，测试用例才会调用此功能。因此，在收集垃圾时，不必太担心竞争条件。在任务1中，我们已经实施了一种算法来计算water mark（系统中最低的read_ts）。在此任务中，您将需要删除所有不包含具有最低read_ts（Watermark）事务的undo log的所有事务。

Garbage Collection并不是将所有ts小于watermark的undolog全部删除，而是要遍历所有事务和undolog进行推导，如上图所示，此时的watermark是3，但是对于Txn9 (read_ts = 3)，当它读取到 (C, 4) 时，由于 ts = 4 > read_ts，所以它要读取undolog找到 ts <= read_ts 的版本，此时只有 (C, 2) ts = 2 能被读取，所以我们不能 collect 该 tuple。

但是不要被lab说明误导，实际上不需要搞清楚watermark下哪些Undolog会被访问，哪些不会，这样会造成大量开销，而且实现起来非常麻烦。我们只需要保持一个watermark可能被访问到的旧版本（该版本应该是所有watermark之上的versionlink所能触及到的watermark之下的一个交集），然后将该旧版本之后的版本删除就可以了，这样就算保留的第一个旧版本这次删除不掉，随着watermark的上移，以后也能被删除掉。

只有COMMIT或者ABORT状态的事务能够被删除。

Task #4 Primary Key Index

Inserts

在 insert executor 中正确处理插入主键的情景，要考虑多个事务插入相同主键的情况。

  1. 首先检查主键已经存在的情况，在后面的delete部分它可能被设置为指向被删除的数据，在此处先将其设置为TAINTED状态。
  2. 然后使用临时时间戳在 table heap 创建元组
  3. 之后，将元组插入索引中。如果违反唯一的密钥约束，则您的哈希索引应返回false。在（1）和（3）之间，其他事务可能正在做同一件事，并且在当前事务创建它之前，在索引中创建了一个新条目。在这种情况下，您将需要中止事务，并且 table heap 中会有一个元组，而没有一个索引指向该元组。

Index Scan, Deletes and Updates

为删除和更新 executor 添加索引支持。首先实现muti-version index scan executor.

一旦在索引中创建了一个条目，即使标记了元组已删除元组，它将始终指向同一RID，也不会删除，因此较早的事务仍然可以使用index scan executor访问历史记录。在这一点上，您还需要修改insert executor。考虑insert executor插入一个已经被删除了的元组的情况。因为索引创建后只会指向同一个RID，所以应更新已删除的元组，而不是创建新的条目。需要正确处理写-写冲突检测和唯一性约束检测。

在此之前delete操作只会出现在undolog链的头部，实现了这一部分后delete可能会出现在undolog的中间位置，所以之前简化的update和delete executor都要修改

此时，您还需要考虑其他竞争条件。例如，如果多个事务正在同时更新版本链接。您应该正确地中止其中一些，让其中一个继续而不会丢失任何数据。在PageVersionInfo中，我们有in_progress字段，该字段指示是否已经有事务在该元组上进行操作。通过此任务，您需要使用此字段来避免竞争条件。另外，您会注意到，在上面的示例中的第二个更改中，table heap中的时间戳有时会与第一个undolog的时间戳相同。实现更新和删除后，Sequential scan executor 还应正确处理此情况。

  • 在封锁之前得获得tuple的时间戳，检查是否是self-modify，如果是，就不需要封锁，因为没有别的事务能访问到该元组，而且之前的操作中也可能已经进行了封锁
  • 但是如果不是self-modify，且未被其他事务修改过，这种情况下简单地直接封锁也会出现问题，因为获取tuple ts的时候没有进行封锁，所以可能会出现多个事务都获取到该tuple ts的情况，而如果时间差非常巧合的话，一个事务封锁并提交解锁后，另一个事务恰好能再次进行封锁，这就导致了一个tuple可能会在同一个read ts被多个txn修改。

目前能想到的方法就是在LockVersionChain后再获取一次ts并进行检查，如果发现不对就要进行abort。在第一次获取ts的时候就加锁肯定是不行的，因为并不知道事务在之前有没有进行加锁，处理起来会非常麻烦
